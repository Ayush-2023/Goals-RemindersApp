Software Requirement Specification
For: Backend API Service of Goals & Reminders(G&R) App(v1)
1 Indroduction
1.1 Purpose

Provide Restful Interface for the G&R App( an app for mananging daily routine tasks) frontend developers.

1.2 Intended Audience and Use

The intended audience of service users include

  • Developers: The API service is designed to serve G&R frontend Developers(mainly website, mobile app)
  • Testers: The API service can be used by testers to test the possibilities of flaws in implementations
  • Genral Users: The API service can be used by general audience to explore the features and suggest improvements to it

1.3 Service Scope

The API Service will provide a Restful Interface to the developers who want to serve those who want to manage their day-to-day tasks, helping them store/retrieve the data from database using API service.

1.4 Definitions and Acronyms

Some terminoligies used for interfacing the API services are as follows:

  • User: A user is a user of G&R App who wants to manage his/her day-to-day tasks
  • Task: A task is a piece of work that can be completed by and be reminded to the users who want to perform it on a particular day. These are characterized by their title
  • Routine Task: Routine Task are peice of work that a user wants to perform after a regualr amount of time, called Repeat Duration(s). They are characterized by thier Task Characterstics, Start Time and Repeat Duration
  • Repeat Duration(s): Repeat Duration(s) is/are amount of time(s) after which the Routine tasks are Assigned to any user. The can be days of a week, dates of months, or dates of years
  • Goal Task: Goal Task are piece of work that a user needs to perform to progress towards achieving his/her Goal. They are characterized by their Task Characterstics and Deadline
  • Scheduled Task: Scheduled Task are piece of work that a user needs to perform on a particular day and time. They are characterized by their Task Characterstics and Scheduled Time
  • Remider: Reminder is notification for the user who has been assigned a particular task.
    For Routine Task reminders are created at Start Time day and on the day, the Repeat Duration Time occurs,
    For Goal Task Remider is created at the time of creation, and
    For Scheduled Task Reminder is created on the scheduled day.

1.5 References

The API service is inspired by general idea of Alarm Clock App found in everyone's Phones. The SRS Doc of Todomoo App was taken as reference for this SRS. Google Tasks App was taken as front-end client reference.

2 Overall Description
2.1 Service Perspective

This API can be used by Developers of G&R App, where they can serve those in need to manage their day-to-day tasks.
The API will be build on Tech Stack of Node.js, Express.js, MongoDB hosted on Render website. It can be directly accessed over internet.
Future Scope of software include paginated implementations of Fetch(READ) Functionalities, caching and google signups.

2.2 User Class and Characterstics

API service can be used by any FrontEnd Developer, API Tester, or any user interested in exploring the features of the Service

2.3 Assumptions and Dependencies

The API service assumes proper working of Node.js, Express.js, and MongoDB technology on respective system. Also assumes proper working of Render Hosting service

3 Service Features and Requirements
3.1 Functional Requirements

  1. Creating New Goals and Task-List
  2. Creating Tasks in their Goals and Task-List
  3. Fetching Goals and Tasks-List
  4. Fetching Task of a particular List
  5. Fetching Task Details(Status of Task)
  6. Editing Goals and Task-List Details
  7. Editing Task Details like Deadline/Schedule/Start Time of a Task
  8. Editing Repeat Duration of a Repetitive Task
  9. Marking a Task to completed
  10. Marking a Remider to be acknowledged
  11. Deleteing Goal and Task-List
  12. Deleting Task of a Goal or Task-List

3.2 External Interface Requirements

  1. Uniform Access accross environments

3.3 Non Functional Requirements

  1. Restful Interface
  2. Persistence of Data accross sessions
  3. Consistency of Data accross sessions
  4. Availability of Service
  5. Efficiency in delivering service
